<!-- Version ID format (must match this exactly) -->
Format: "{ISO8601-ms}-{UUID-prefix-8}"
  ISO8601-ms: DateFormatter with "yyyy-MM-dd'T'HH:mm:ss.SSS" format, UTC timezone
  UUID-prefix: UUID().uuidString.prefix(8).lowercased()
  Example: "2026-02-25T143022.456-a3f8b12c"

Lexicographic sort = chronological sort (ISO8601 is designed for this).
Two calls within the same millisecond still produce different IDs (UUID suffix).

<!-- Pruning policy (from research open questions resolution) -->
Corrupt versions: Do NOT count toward retention limit. Keep in DB with status=corrupt.
  Rationale: User may want to inspect why a version was corrupt. Surface in history UI (Phase 3)
  with a warning indicator. Re-backup should be triggered (BackupEngine responsibility, not VersionManager).

deleting status: Write-then-cleanup pattern.
  VersionManager marks status='deleting' inside a DB write transaction BEFORE touching disk.
  After transaction commits, delete from disk.
  On crash between mark and delete: 'deleting' rows are re-processed on next launch.
  BackupEngine is responsible for cleanup of 'deleting' rows at startup.

Locked versions: VersionLock table (schema from 01-01).
  If a version has a VersionLock row, skip it during pruning.
  Used in Phase 3 (restore). VersionManager should respect locks in Phase 1.

retentionCount: Read from DestinationConfig.retentionCount (default 10).
  Apply per (projectID + destinationID) pair — each destination has its own retention.
  Reason: a local drive and a NAS may have different capacities.

    Pruning (pruneOldVersions):
    - Given projectID, destinationID, retentionCount, and a DatabasePool:
      - Fetch all verified versions for (projectID, destinationID), ordered by createdAt ASC
      - Count = verified versions only (corrupt versions excluded from count)
      - Excess = all verified versions beyond the last {retentionCount} (oldest first)
      - Skip any version that has an active VersionLock
      - Mark remaining excess versions status='deleting' in a single DB write transaction
      - After transaction: delete version directories from disk (caller provides rootPath)
      - Return list of deleted version IDs
