<decisions_made>

- Swift 6 + SwiftUI/AppKit hybrid (MenuBarExtra + NSStatusItem)
- GRDB 7 with DatabasePool (WAL mode, concurrent reads) — not Core Data
- COPYFILE_CLONE (not COPYFILE_CLONE_FORCE) — preserves ExFAT compatibility for audio drives
- xxHash3 for checksums (15x faster than SHA-256 on Apple Silicon); CryptoKit SHA-256 as fallback
- mtime+size for incremental skip decisions (not source checksum — would defeat incremental)
- Version lifecycle state machine: pending → copying → copy_complete → verifying → verified | corrupt
- Corrupt versions excluded from retention count (kept for inspection, not silently deleted)
- Write-then-cleanup pruning (deleting status set in DB before disk deletion)
- v1 destinations: Local, NAS (SMB/NFS), iCloud Drive, GitHub+LFS (Google Drive + Dropbox deferred to v2)
- ALS parser included in v1 (Phase 5) — it's the core differentiator
- Distribute outside Mac App Store initially (avoid sandbox restrictions on FSEvents, SMB, git shell)

</decisions_made>

<context>
This is a greenfield macOS menu bar app. The key insight from research: the ALS parser (parsing gzipped XML .als files to find externally-referenced samples) is the core moat — every general-purpose backup tool treats the Ableton project folder as just a folder, silently missing samples stored elsewhere. Phase 1 builds the pure Swift backup engine with no UI — a tested, verified foundation before layering the app shell on top in Phase 2.

The 6-phase roadmap:
1. Backup Engine (Swift package, no UI) ← we are here, ready to execute
2. App Shell + Triggers (menu bar, FSEvents, schedule, notifications)
3. Settings + History (settings window, version browser)
4. Network Destinations (NAS SMB/NFS, iCloud Drive)
5. ALS Parser (external sample resolution)
6. GitHub Destination (Git LFS, bundled binaries)
</context>
